Correctness Properties
para el RAG

Invariantes que tu código DEBE cumplir siempre, sin importar el input. No son tests de un caso — son leyes universales de tu sistema que garantizan que el RAG nunca miente, nunca inventa y siempre cita sus fuentes.

Explorar →
01 — Concepto fundamental

¿Qué son las Correctness Properties?

Una Correctness Property es una invariante que el código DEBE cumplir siempre, para cualquier input válido. No es un test de un caso específico — es una ley universal de tu sistema. Si se viola, hay un bug. Punto.

"Un property test no verifica que la función retorna 42 para el input 7. Verifica que para TODO input posible, la función cumple una propiedad: el resultado siempre es positivo, siempre es menor que el input, siempre contiene al menos un elemento."

— Concepto central de Property-Based Testing
🔬

Analogía: Leyes de la física

La gravedad siempre atrae objetos hacia abajo — no solo los martes ni solo con pelotas rojas. Una correctness property se cumple SIEMPRE, para TODOS los inputs. Es una ley de tu sistema.

🆚

Property vs Example Test

Example test: "para input 'hola', retorna 'HOLA'".
Property: "para TODO input string, el output tiene la misma longitud que el input". Miles de casos en una sola declaración.

🎯

¿Por qué importa para RAG?

Un RAG maneja inputs impredecibles (preguntas de usuarios). No puedes testear todas las preguntas posibles — pero sí puedes garantizar propiedades universales: "siempre cita fuentes", "nunca inventa".

🏗️

En el design.md de Kiro

Las correctness properties viven en el design.md como invariantes que cada task debe respetar. El agente las verifica automáticamente al completar cada tarea.

02 — Propiedades universales del RAG

4 invariantes que tu RAG DEBE cumplir siempre

Estas son las propiedades de corrección fundamentales para cualquier RAG sobre documentación interna. Si alguna se viola, el sistema no es confiable para uso en producción.

1
Citación obligatoria de fuentes

Para toda consulta que genere una respuesta, el RAG SIEMPRE cita las fuentes de donde extrajo la información.

∀ query ∈ Consultas, ∀ response = RAG(query): response.sources.length ≥ 1 ∧ ∀ source ∈ response.sources: source.documentName ∈ DocumentosIndexados ∧ source.chunk ≠ ""
⚠️ Si se viola

El usuario recibe respuestas sin respaldo. No puede verificar si la información es correcta. Pierde confianza en el sistema. En contexto regulatorio (AFP UNO / Superintendencia), esto puede tener consecuencias legales.

✅ Cómo se verifica

Para N consultas aleatorias generadas, verificar que response.sources.length >= 1 y que cada source referencia un documento que existe en el index.

2
No fabricación (No Hallucination)

El RAG NUNCA fabrica información que no existe en la Knowledge Base. Toda afirmación en la respuesta debe ser rastreable a un chunk indexado.

∀ query ∈ Consultas, ∀ response = RAG(query): ∀ claim ∈ response.claims: ∃ chunk ∈ IndexedChunks: semanticSimilarity(claim, chunk) ≥ GROUNDING_THRESHOLD
⚠️ Si se viola

El RAG "inventa" información. En AFP UNO, podría generar montos de cotizaciones falsos, normativas inexistentes o procedimientos incorrectos. Un RAG que alucina es peor que no tener RAG.

✅ Cómo se verifica

Para cada afirmación en la respuesta, calcular similitud semántica con los chunks usados. Si alguna afirmación no tiene anclaje suficiente (score < threshold), el property falla.

3
Respuesta acotada (Bounded Response)

La longitud de la respuesta está acotada — el RAG no genera respuestas infinitas ni desproporcionadamente largas.

∀ query ∈ Consultas, ∀ response = RAG(query): response.answer.length ≤ MAX_RESPONSE_LENGTH ∧ response.answer.length > 0 ∧ response.processingTime ≤ MAX_TIMEOUT
⚠️ Si se viola

Costos de API se disparan (más tokens = más caro). UX degradada (usuario espera demasiado). Posible timeout. En batch processing, una respuesta infinita bloquea todo el pipeline.

✅ Cómo se verifica

Para N consultas aleatorias: response.length <= MAX, response.length > 0, y processingTime <= timeout. Incluir inputs extremos: consultas muy largas, vacías, en otros idiomas.

4
Umbral de Grounding configurable

El score de grounding (anclaje a fuentes) cumple un umbral mínimo configurable. Si no se alcanza, el sistema lo declara explícitamente en vez de responder con baja confianza.

∀ query ∈ Consultas, ∀ response = RAG(query): IF max(chunk_scores) < GROUNDING_THRESHOLD: response.type = "NO_RELEVANT_INFO" ∧ response.answer CONTAINS "no encontré información" ELSE: response.confidence ≥ GROUNDING_THRESHOLD ∧ response.sources[0].score ≥ GROUNDING_THRESHOLD
⚠️ Si se viola

El RAG responde con baja confianza sin advertir. El usuario asume que la respuesta es confiable cuando en realidad no tiene buen anclaje a fuentes. Decisiones basadas en información poco fiable.

✅ Cómo se verifica

Generar consultas que NO tienen respuesta en la KB (temas no cubiertos). Verificar que el sistema retorna "no encontré información" en vez de inventar. Verificar que el threshold es configurable externamente.

03 — Clasificación de criterios

Property vs Example vs Edge-Case

No todo acceptance criteria es un property. Al hacer prework analysis de requirements, debes clasificar cada criterio en una de estas 3 categorías. Esto determina cómo se testea.

🔬

Property

Invariante universal. Se cumple para TODOS los inputs posibles. Se testea con generación aleatoria de inputs (PBT).

Ejemplo: "Toda respuesta tiene sources.length ≥ 1"

📝

Example

Un caso concreto y específico. Se cumple para UN input determinado. Se testea con unit tests tradicionales.

Ejemplo: "Para 'procedimiento cotizaciones', retorna info del doc X"

⚡

Edge-Case

Caso de borde o situación extrema. Comportamiento ante inputs inusuales, vacíos, malformados o límites del sistema.

Ejemplo: "Consulta vacía retorna error 400 con mensaje claro"

Ejercicio de clasificación: dado estos acceptance criteria, ¿cuál es cuál?

AC-1: "El RAG siempre retorna respuesta en menos de 5s" → PROPERTY (para TODO input, tiempo < 5s) AC-2: "Para la pregunta '¿qué es una AFP?', retorna definición del glosario" → EXAMPLE (un caso específico, un input concreto) AC-3: "Si el usuario envía un string de 10,000 caracteres, el sistema lo trunca" → EDGE-CASE (input extremo, límite del sistema) AC-4: "Toda respuesta incluye al menos una fuente citada" → PROPERTY (para TODA respuesta, sin excepción) AC-5: "Si la conexión a Bedrock falla, retorna error 503" → EDGE-CASE (situación de fallo, no normal) AC-6: "Para 'procedimiento deploy', retorna pasos del doc guia-deploy.md" → EXAMPLE (caso específico con input y output conocido)
🔬

¿Cómo testear properties?

Con Property-Based Testing (PBT): genera cientos de inputs aleatorios y verifica que la propiedad se cumple para TODOS. Si falla para uno solo, hay bug.

📝

¿Cómo testear examples?

Con unit tests tradicionales: input fijo → output esperado. Son los tests que ya conoces: expect(result).toBe(expected).

⚡

¿Cómo testear edge-cases?

Con tests específicos de boundary: inputs vacíos, nulos, extremadamente largos, caracteres especiales, timeouts, desconexiones.

04 — Verificación formal de acceptance criteria

Consistency, Completeness, Conflicts

Antes de implementar, los acceptance criteria deben pasar una verificación formal. ¿Se contradicen? ¿Cubren todos los casos? ¿Hay conflictos? Kiro puede detectar estos problemas automáticamente al analizar los requirements.

🔄

Consistency (Consistencia)

Los criteria no se contradicen entre sí. Si AC-1 dice "responder siempre en < 5s" y AC-7 dice "procesar indexaciones masivas sin límite de tiempo", hay una contradicción potencial si la indexación bloquea consultas.

Pregunta clave: ¿Puedo cumplir TODOS los criteria simultáneamente sin que uno viole a otro?

📋

Completeness (Completitud)

Los criteria cubren todos los escenarios posibles. ¿Qué pasa si el input es vacío? ¿Y si Bedrock está caído? ¿Y si el documento indexado se eliminó después de indexar? Si hay un escenario sin criterio definido, el comportamiento es impredecible.

Pregunta clave: ¿Hay algún input o situación posible que no está cubierta por ningún criteria?

⚔️

Conflicts (Conflictos)

Dos criteria exigen comportamientos opuestos ante el mismo input. Ejemplo: AC-3 dice "si no hay resultados, retornar mensaje vacío" y AC-5 dice "toda respuesta debe incluir al menos una sugerencia". ¿Qué pasa cuando no hay resultados? Conflicto.

Pregunta clave: ¿Hay dos criteria que aplican al mismo input pero exigen respuestas diferentes?

Ejemplo: Detección de conflicto en Kiro

# Kiro analiza tus requirements y detecta: ⚠️ CONFLICTO detectado entre REQ-3 y REQ-7: REQ-3 [Event-Driven]: "WHEN un usuario consulta, el servicio SHALL retornar respuesta con al menos 1 fuente citada." REQ-7 [Unwanted Behavior]: "IF no hay documentos relevantes, THEN el servicio SHALL retornar 'no encontré información'." → CONFLICTO: Si no hay documentos relevantes, ¿retorna con fuente citada (REQ-3) o sin fuente (REQ-7)? 💡 SUGERENCIA: Agregar excepción a REQ-3: "...con al menos 1 fuente citada, EXCEPTO cuando aplique REQ-7."

Regla: Toda inconsistencia, incompletitud o conflicto en los acceptance criteria se convierte en un bug en producción. Es más barato detectarlos en la spec que en el código.
