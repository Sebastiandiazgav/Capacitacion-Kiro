# 3.3 Workshop: PBT para el RAG Evolutivo + Bugfix Workflow

## Objetivo

Escribir las 4 correctness properties formales para el RAG, ejecutar PBT con 100+ iteraciones para encontrar edge cases, y aplicar el bugfix workflow completo sobre el bug encontrado.

**Tiempo:** 45 minutos  
**Prerrequisitos:** RAG con EARS (M2.3), fast-check instalado (M3.2), Correctness Properties (M3.1)

---

## Ejercicio 1: Escribir las 4 properties del RAG con Kiro (20 min)

### Paso 1.1: Prompt al agente

Abre Kiro en modo Vibe y pega este prompt:

```
Tengo un RAG especificado con EARS en .kiro/specs/rag-knowledge-base/.
Necesito que generes Property-Based Tests con fast-check para estas 4 correctness properties:

PROPERTY 1 — Citación obligatoria:
"Para toda consulta válida (string no vacío, <= 500 chars), el RAG siempre
cita al menos una fuente existente en la KB cuando retorna tipo ANSWER."

PROPERTY 2 — No fabricación:
"Para toda respuesta generada, ningún fragmento en sources[] contiene
información que no está presente en los documentos indexados. El chunk
citado DEBE existir literalmente en el documento referenciado por documentName."

PROPERTY 3 — Respuesta acotada:
"Para toda respuesta, la longitud del campo answer no excede 3000 caracteres
y es mayor a 0. El tiempo de procesamiento no excede 10 segundos."

PROPERTY 4 — Umbral de grounding:
"Para toda consulta, si el mejor score de similitud es menor a 0.3,
el sistema retorna type NO_RELEVANT_INFO. Si retorna type ANSWER,
el score del primer source es >= 0.3."

ADICIONAL — Genera generadores de:
- Consultas aleatorias variadas (normales, con ñ/tildes, emojis, muy largas)
- Documentos sintéticos para indexar (variada longitud, encoding UTF-8)
- Consultas irrelevantes (temas no cubiertos por la KB)

Configura numRuns: 100 para cada property.
Usa vitest como runner.
Genera todo en tests/properties/
```

### Paso 1.2: Revisar y ajustar

Verifica que Kiro generó:
- [ ] `tests/properties/generators.ts` — generadores variados
- [ ] `tests/properties/rag-properties.test.ts` — 4 property tests
- [ ] Cada test usa `numRuns: 100`
- [ ] Los imports referencian correctamente tu función `queryRAG`

### Paso 1.3: Ejecutar las 4 properties

```bash
npx vitest run tests/properties/
```

Observa:
- ¿Pasan todas las properties?
- Si alguna falla → ¿cuál es el input que la viola? (ver shrinking)
- Anota el edge case encontrado → lo usaremos en el Ejercicio 2

## Ejercicio 2: Bugfix Workflow completo (25 min)

### Paso 2.1: Identificar el bug (5 min)

Del Ejercicio 1, al menos una property debería haber encontrado un edge case. Posibles bugs:

- La property de "no fabricación" falla porque el chunk no coincide con el documentName
- La property de "respuesta acotada" falla con consultas muy largas
- La property de "grounding" falla porque retorna ANSWER con score < 0.3

Si todas pasan (raro con 100 iteraciones), provoca un fallo:
```typescript
// Temporalmente en tu RAG, introduce este bug:
// En el retriever, mezcla aleatoriamente el documentName con otro doc
if (Math.random() > 0.9) {
  source.documentName = "documento-incorrecto.md";
}
```

### Paso 2.2: Reproducir el bug

El input que PBT encontró (con shrinking) es tu test de reproducción:

```typescript
// Test que reproduce el bug exacto
it('Reproduce bug: fuente incorrecta', async () => {
  const bugInput = "INPUT_QUE_PBT_ENCONTRO"; // copiar del output de shrinking
  const response = await queryRAG(bugInput);
  // Este test DEBE fallar antes del fix
  expect(response.sources[0].documentName).not.toBe("documento-incorrecto.md");
});
```

### Paso 2.3: Crear Bugfix Spec en Kiro (5 min)

En el chat de Kiro, pide crear un Bugfix Spec:

```
Necesito crear un Bugfix Spec para el siguiente bug:

BUG: El RAG a veces cita una fuente incorrecta — el documentName en sources[]
no corresponde al documento real del cual se extrajo el chunk.

CURRENT BEHAVIOR (lo que hace ahora — MAL):
- Para ciertos inputs, el campo sources[].documentName referencia un documento
  que no contiene el chunk citado en sources[].chunk.
- Esto ocurre cuando [describir la causa raíz que encontraste].

EXPECTED BEHAVIOR (lo que debería hacer):
- Para toda respuesta, cada sources[].chunk DEBE existir literalmente en el
  contenido del archivo referenciado por sources[].documentName.

UNCHANGED BEHAVIOR (lo que NO debe cambiar):
- Tiempo de respuesta
- Ranking de relevancia
- Indexación de documentos
- Las properties CP-1, CP-3, CP-4 deben seguir pasando

Crea la bugfix spec y genera tasks para el fix.
```

### Paso 2.4: Implementar el fix (10 min)

Kiro generará tasks desde la bugfix spec. Ejecuta las tasks:
1. El agente corrige el mapeo chunk → documentName
2. Verifica que la metadata se asigna durante el split, no después
3. Ejecuta los tests existentes para no-regression

### Paso 2.5: Verificar con PBT (5 min)

Ejecuta nuevamente TODAS las properties:

```bash
npx vitest run tests/properties/
```

Verifica:
- [ ] La property que fallaba ahora PASA (bug corregido)
- [ ] Todas las demás properties siguen pasando (no regression)
- [ ] El test de reproducción del bug pasa
- [ ] 100 iteraciones completas sin fallos
