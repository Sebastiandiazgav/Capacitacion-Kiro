Detectar ambigüedades y conflictos en requisitos ANTES de implementar, y cuando un bug aparece, resolverlo con un ciclo riguroso: reproducir → spec → fix → verificar con PBT. Sin regresiones.

01 — Análisis de ambigüedades

Requisitos que parecen claros pero no lo son

Un requisito puede parecer correcto a simple vista pero ocultar ambigüedades que solo se manifiestan durante la implementación. La verificación formal detecta estos problemas antes de escribir código.

Ejemplo: Ambigüedad oculta en un requisito del RAG

# Requisito aparentemente claro: "WHEN un usuario consulta el RAG, el sistema SHALL responder con fuentes citadas en menos de 5 segundos." # Ambigüedades ocultas: ❓ ¿"responder con fuentes" incluye el caso donde no hay matches? ❓ ¿"menos de 5 segundos" es tiempo total o solo tiempo de LLM? ❓ ¿"fuentes citadas" = nombre del archivo, o nombre + chunk + score? ❓ ¿Si Bedrock tarda 4.9s y la post-procesión 0.2s, cumple o no? # Versión sin ambigüedad: "WHEN un usuario envía una consulta válida (string no vacío, longitud <= 500 chars) al endpoint /query, el servicio de consultas SHALL retornar un JSON con campo 'sources' conteniendo al menos 1 objeto {documentName, chunk, score} donde score >= 0.3, con tiempo total de respuesta (incluyendo red) < 5000ms medido en p95 sobre ventana de 1 minuto."
🔍

Palabras peligrosas

"rápido", "correcto", "adecuado", "relevante", "apropiado" — todas son ambiguas. Reemplazar por métricas: ¿cuántos ms? ¿qué score? ¿qué formato exacto?

🎯

Test de la pregunta "¿y si...?"

Por cada requisito, pregunta: "¿y si el input es vacío?", "¿y si hay timeout?", "¿y si hay 0 matches?". Si no sabes la respuesta → hay ambigüedad.

🤖

Kiro detecta automáticamente

Al crear una spec, Kiro señala requirements con métricas faltantes, condiciones implícitas o respuestas vagas. Te sugiere la versión desambiguada.

02 — Detección de conflictos

Cuando dos criterios se contradicen

Dos acceptance criteria pueden exigir comportamientos opuestos ante el mismo input. Estos conflictos se convierten en bugs inevitables si no se detectan en la fase de requisitos.

Ejemplo: Conflicto real en el RAG

# CONFLICTO entre REQ-3 y REQ-7: REQ-3 [Event-Driven]: "WHEN un usuario consulta, el servicio SHALL retornar respuesta con al menos 1 fuente citada." REQ-7 [Unwanted Behavior]: "IF no hay documentos relevantes (score < 0.3), THEN el servicio SHALL retornar 'No encontré información'." ⚔️ CONFLICTO: Cuando no hay matches relevantes... - REQ-3 exige: respuesta CON fuente - REQ-7 exige: respuesta SIN fuente (solo mensaje) 💡 RESOLUCIÓN: Agregar excepción explícita a REQ-3: "...con al menos 1 fuente citada, EXCEPTO cuando aplique la condición de REQ-7 (sin matches relevantes)."
⚔️

Cómo detectar conflictos

Para cada par de requirements, pregunta: "¿Existe algún input para el cual ambos aplican pero exigen respuestas diferentes?" Si sí → conflicto.

🔧

Cómo resolver

Priorización: un requirement gana sobre otro. Excepción: "EXCEPTO cuando...". División: separar en sub-escenarios mutuamente excluyentes.

🧪

Conflicto → Property Test

Todo conflicto resuelto se convierte en un property test: "Para inputs en la zona de conflicto, el sistema sigue la regla X (no la Y)". PBT verifica con inputs aleatorios.

03 — Bugfix Workflow

Reproducir → Spec → Fix → Verify

Cuando un bug aparece, no lo "arreglas directo". Sigues un ciclo riguroso que documenta el problema, previene regresiones y garantiza que el fix es correcto. Kiro tiene Bugfix Specs nativos para esto.

Ciclo Bugfix Workflow

🐛 Bug detectado
→
🔬 Reproducir
→
📋 Bugfix Spec
→
🔧 Implementar Fix
→
✅ Verificar con PBT
🔬

1. Reproducir

Encontrar el input exacto que causa el bug. Idealmente con un test que falla consistentemente. Si PBT lo encontró → ya tienes el input mínimo (shrinking).

📋

2. Bugfix Spec

Crear spec en Kiro (tipo Bugfix). Documenta:
• Current behavior: qué hace ahora (mal)
• Expected behavior: qué debería hacer
• Unchanged behavior: qué NO debe cambiar

🔧

3. Implementar Fix

Kiro genera tasks desde la bugfix spec. El fix se implementa con contexto completo: sabe qué NO tocar (unchanged behavior) y qué debe cambiar.

✅

4. Verificar con PBT

Ejecutar property tests para confirmar:
• La property que se violaba ahora pasa
• Ninguna otra property se rompió (no regression)
• El input que causó el bug ya no falla

Bugfix Spec en Kiro (estructura)

# .kiro/specs/fix-fuente-incorrecta/bugfix.md ## Bug: RAG cita fuente incorrecta ### Current Behavior (comportamiento actual — MAL): Cuando el RAG procesa una consulta que coincide con múltiples documentos, el campo documentName en sources[] a veces referencia un documento diferente al que contiene el chunk citado. ### Expected Behavior (comportamiento esperado — BIEN): Para toda respuesta, cada objeto en sources[] debe tener un documentName que corresponde exactamente al archivo del cual se extrajo el chunk citado. ### Unchanged Behavior (no debe cambiar): - El tiempo de respuesta no debe aumentar - El ranking de relevancia no debe alterarse - La función de indexación no debe modificarse - Las demás properties (CP-1, CP-3, CP-4) deben seguir pasando

¿Por qué no "arreglar directo"? Sin bugfix spec: arreglas el bug pero introduces un regression en otro lado. Con bugfix spec: documentas qué NO tocar, el agente respeta los boundaries, y PBT verifica que no rompiste nada.

04 — Caso práctico

Bug: RAG cita fuente incorrecta

Escenario real: un usuario consulta sobre "procedimiento de cotizaciones" y el RAG responde correctamente, pero el campo documentName dice "guia-deploy-microservicios.md" cuando el chunk en realidad viene de "procedimiento-cotizaciones.md".

Flujo del bug

LO QUE PASA (MAL)
❓ Query: "cotizaciones"
→
Retriever: chunk correcto
→
❌ Metadata: doc incorrecto
→
📄 Fuente: "guia-deploy.md"
LO QUE DEBERÍA PASAR (BIEN)
❓ Query: "cotizaciones"
→
Retriever: chunk correcto
→
✅ Metadata: doc correcto
→
📄 Fuente: "procedimiento-cotizaciones.md"
🐛

Causa raíz

El text splitter genera chunks y les asigna metadata con el nombre del documento. Pero cuando dos documentos se procesan en paralelo, el índice de metadata se desfasa del índice de chunks.

🧪

Property que lo detecta

CP-2: "Para toda fuente citada en sources[], el chunk DEBE existir literalmente en el documento referenciado por documentName." PBT encuentra el fallo con múltiples docs.

🔧

Fix

Asignar metadata al chunk DURANTE el split (no después). Cada chunk lleva su documentName embebido desde el momento de creación, no por posición en array.
