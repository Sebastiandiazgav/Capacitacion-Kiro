# 3.2 Workshop: PBT en Acción

## Objetivo

Implementar Property-Based Tests sobre el RAG usando fast-check. Primero usando Kiro para generar los tests automáticamente, luego entendiendo y ejecutando los tests manualmente.

**Tiempo:** 45 minutos  
**Herramienta PBT:** fast-check (TypeScript)  
**Prerrequisitos:** RAG funcional de M1.3, comprensión de Correctness Properties (M3.1)

---

## Ejercicio 1: Generar PBT con Kiro (15 min)

### Objetivo
Darle a Kiro un prompt estructurado con las correctness properties y que el agente genere automáticamente los property tests con fast-check.

### Paso 1.1: Abrir Kiro en modo Vibe

Abre el proyecto del RAG en Kiro y abre el chat (modo Vibe).

### Paso 1.2: Prompt al agente

Copia y pega este prompt en el chat de Kiro:

```
Necesito que generes Property-Based Tests con fast-check para mi RAG.

CONTEXTO DEL RAG:
- Función principal: queryRAG(question: string)
- Retorna: { answer: string, sources: Array<{documentName: string, chunk: string, score: number}>, type: string, confidence: number }
- Tipos de type: "ANSWER" (respuesta normal) o "NO_RELEVANT_INFO" (sin matches)
- Los documentos están en la carpeta docs/ como archivos .md
- Usa Bedrock con Claude Sonnet + Titan Embeddings

CORRECTNESS PROPERTIES A TESTEAR:
1. CP-1 (Citación obligatoria): Toda respuesta con type "ANSWER" incluye al menos 1 fuente citada (sources.length >= 1) y cada fuente tiene documentName no vacío.
2. CP-2 (No fabricación): Toda fuente citada referencia un documento que existe en la carpeta docs/.
3. CP-3 (Respuesta acotada): La respuesta siempre tiene longitud <= 3000 caracteres y >= 0 caracteres. Nunca null ni undefined.
4. CP-4 (Umbral de grounding): Si ningún chunk tiene score >= 0.3, el type debe ser "NO_RELEVANT_INFO".

CASOS ADICIONALES:
- Consulta vacía ("", " ", "\n") → no lanza excepción, retorna gracefully
- Consultas con caracteres especiales (ñ, tildes, emojis) → no falla
- Misma consulta 2 veces → mismo ranking de fuentes (determinismo)

GENERA:
1. Archivo tests/generators.ts con generadores custom (arbitraries) para:
   - Consultas realistas del dominio AFP UNO
   - Consultas aleatorias
   - Consultas con caracteres especiales y edge cases
   - Mix ponderado de todos (fc.oneof con weights)

2. Archivo tests/rag.property.test.ts con:
   - 4 property tests (uno por cada CP)
   - 3 tests de casos adicionales
   - Configuración: numRuns: 100 para cada property
   - Usar vitest como runner

3. Instalar las dependencias necesarias (fast-check, vitest)
```

### Paso 1.3: Revisar lo que Kiro generó

Después de que Kiro genere los archivos, revisa:

- [ ] ¿Creó `tests/generators.ts` con generadores variados?
- [ ] ¿Creó `tests/rag.property.test.ts` con los 4 CP + 3 adicionales?
- [ ] ¿Instaló fast-check y vitest?
- [ ] ¿Los tests referencian correctamente la función `queryRAG`?
- [ ] ¿Usa `numRuns: 100`?

### Paso 1.4: Ajustar si es necesario


### Paso 2.2: Crear generadores custom (10 min)

Crea el archivo `tests/generators.ts`:

```typescript
import fc from 'fast-check';

// Consultas normales (simulan usuarios reales)
export const realQueryArbitrary = fc.constantFrom(
  "¿Cuál es el procedimiento para cotizaciones?",
  "¿Cómo hago deploy de un microservicio?",
  "¿Qué dice la normativa sobre fondos?",
  "¿Cuál es el proceso de escalamiento?",
  "¿Cómo reporto un incidente?"
);

// Consultas aleatorias (strings random)
export const randomQueryArbitrary = fc.string({ minLength: 1, maxLength: 500 });

// Consultas con caracteres especiales
export const specialCharsQueryArbitrary = fc.unicode({ minLength: 1, maxLength: 200 });

// Consultas vacías o edge-case
export const edgeCaseQueryArbitrary = fc.constantFrom(
  "",
  " ",
  "\n",
  "\t\t\t",
  "a".repeat(5000),
  "🧪🔬💡",
  "<script>alert('xss')</script>"
);

// Mix de todo: realistas + aleatorias + extremas
export const queryArbitrary = fc.oneof(
  { weight: 3, arbitrary: realQueryArbitrary },
  { weight: 4, arbitrary: randomQueryArbitrary },
  { weight: 2, arbitrary: specialCharsQueryArbitrary },
  { weight: 1, arbitrary: edgeCaseQueryArbitrary }
);

// Documentos de longitud variable
export const documentArbitrary = fc.record({
  name: fc.string({ minLength: 3, maxLength: 30 }).map(s => s.replace(/[/\\]/g, '') + '.md'),
  content: fc.string({ minLength: 50, maxLength: 5000 }),
});
```

### Paso 2.3: Escribir property tests (15 min)

Crea el archivo `tests/rag.property.test.ts`:

```typescript
import { describe, it } from 'vitest';
import fc from 'fast-check';
import { queryRAG } from '../src/rag'; // ajustar al path real
import { queryArbitrary, edgeCaseQueryArbitrary } from './generators';

describe('Correctness Properties del RAG', () => {

  // CP-1: Toda respuesta incluye al menos una fuente citada
  it('CP-1: Toda respuesta con contenido incluye fuentes', async () => {
    await fc.assert(
      fc.asyncProperty(queryArbitrary, async (query) => {
        const response = await queryRAG(query);
        if (response.type === 'NO_RELEVANT_INFO') return true;
        return response.sources.length >= 1
            && response.sources.every(s => s.documentName !== '');
      }),
      { numRuns: 100 }
    );
  });

  // CP-3: Respuesta siempre acotada
  it('CP-3: Respuesta nunca excede longitud máxima', async () => {
    await fc.assert(
      fc.asyncProperty(queryArbitrary, async (query) => {
        const response = await queryRAG(query);
        return response.answer.length <= 3000
            && response.answer.length >= 0;
      }),
      { numRuns: 100 }
    );
  });

  // CP-4: Sin consultas relevantes → declara explícitamente
  it('CP-4: Consultas sin match retornan NO_RELEVANT_INFO', async () => {
    const irrelevantQuery = fc.constantFrom(
      "receta de pizza margarita",
      "resultado del partido de fútbol",
      "clima en Tokio mañana"
    );
    await fc.assert(
      fc.asyncProperty(irrelevantQuery, async (query) => {
        const response = await queryRAG(query);
        return response.type === 'NO_RELEVANT_INFO'
            || response.confidence >= 0.3;
      }),
      { numRuns: 50 }
    );
  });

  // CASO 1: Consulta vacía — graceful handling
  it('Consulta vacía no lanza excepción', async () => {
    await fc.assert(
      fc.asyncProperty(edgeCaseQueryArbitrary, async (query) => {
        try {
          const response = await queryRAG(query);
          // No lanzó excepción — OK
          return response !== null && response !== undefined;
        } catch (e) {
          // Si lanza excepción → property violada
          return false;
        }
      }),
      { numRuns: 50 }
    );
  });

  // CASO 3: Ranking estable
  it('Dos consultas iguales retornan el mismo ranking', async () => {
    await fc.assert(
      fc.asyncProperty(
        fc.constantFrom(
          "procedimiento cotizaciones",
          "deploy microservicio",
          "normativa fondos"
        ),
        async (query) => {
          const r1 = await queryRAG(query);
          const r2 = await queryRAG(query);
          if (r1.sources.length === 0) return true;
          // El primer source debe ser el mismo
          return r1.sources[0].documentName === r2.sources[0].documentName;
        }
      ),
      { numRuns: 30 }
    );
  });
});
```
