01 — ¿Qué es Property-Based Testing?

De example-based a property-based

En testing tradicional (example-based), tú defines inputs y outputs esperados. En PBT, defines una PROPIEDAD que siempre debe cumplirse y el framework genera inputs aleatorios para encontrar violaciones. Es la forma de testear lo que no puedes anticipar.

📝 Example-Based Testing (tradicional)

// Tú defines cada caso manualmente test("suma 2+3 = 5", () => { expect(suma(2, 3)).toBe(5); }); test("suma 0+0 = 0", () => { expect(suma(0, 0)).toBe(0); }); // ¿Y si suma(-1, MAX_INT) falla? // No lo sabes hasta que alguien lo descubre

🔬 Property-Based Testing

// Defines UNA propiedad universal test.prop([fc.integer(), fc.integer()])( "suma es conmutativa", (a, b) => { expect(suma(a, b)).toBe(suma(b, a)); } ); // fast-check genera 100+ pares aleatorios // y verifica que la propiedad se cumple SIEMPRE // Si falla con (-1, MAX_INT) → te lo dice
🎲

Inputs aleatorios

El framework genera cientos de inputs que tú nunca habrías pensado: strings vacíos, números negativos, caracteres Unicode, longitudes extremas.

🔍

Encuentra lo inesperado

PBT descubre edge cases que los humanos no anticipamos. Bugs que solo aparecen con cierta combinación de inputs que nadie habría escrito como test.

📐

Menos tests, más cobertura

Una sola property con 100 iteraciones cubre más escenarios que 20 unit tests manuales. Define la regla, el framework hace el trabajo.

02 — Librerías PBT

fast-check & Hypothesis

Las dos librerías principales de PBT en los ecosistemas que usamos. Para este workshop usaremos fast-check (TypeScript), pero Hypothesis es la referencia en Python.

⚡ fast-check (TypeScript/JS)

// Instalación npm install fast-check --save-dev // Uso básico import fc from 'fast-check'; fc.assert( fc.property( fc.string(), // generador de strings aleatorios (input) => { // La propiedad que siempre debe cumplirse const result = miFuncion(input); return result.length <= MAX_LENGTH; } ), { numRuns: 100 } // 100 iteraciones );

Muy buena integración con Jest/Vitest. Shrinking automático. TypeScript nativo. Generadores composables.

🐍 Hypothesis (Python)

# Instalación pip install hypothesis # Uso básico from hypothesis import given from hypothesis import strategies as st @given(st.text()) def test_mi_funcion(input_text): """La propiedad que siempre debe cumplirse""" result = mi_funcion(input_text) assert len(result) <= MAX_LENGTH

El estándar de PBT en Python. Decorador @given elegante. Database de ejemplos que fallan. Integración con pytest.

03 — Generadores personalizados

Arbitraries: inputs a medida

Los generadores (llamados "arbitraries" en fast-check) producen datos aleatorios con la forma que necesitas. Puedes componer generadores simples para crear inputs complejos específicos de tu dominio.

Generadores para el RAG de AFP UNO

import fc from 'fast-check'; // Generador de consultas aleatorias al RAG const queryArbitrary = fc.oneof( fc.string({ minLength: 1, maxLength: 500 }), // strings normales fc.constantFrom( // consultas reales "¿Cuál es el procedimiento de cotizaciones?", "¿Cómo hago deploy?", "¿Qué dice la normativa sobre fondos?" ), fc.string({ minLength: 0, maxLength: 0 }), // consulta vacía fc.unicode({ minLength: 1, maxLength: 200 }), // con caracteres especiales fc.stringOf(fc.constantFrom('á','é','í','ñ','ü')), // solo acentos/ñ ); // Generador de documentos de longitud variable const documentArbitrary = fc.record({ name: fc.string({ minLength: 3, maxLength: 50 }).map(s => s + '.md'), content: fc.string({ minLength: 10, maxLength: 10000 }), encoding: fc.constantFrom('utf-8', 'ascii', 'latin1'), }); // Generador de configuración del RAG const configArbitrary = fc.record({ chunkSize: fc.integer({ min: 100, max: 2000 }), chunkOverlap: fc.integer({ min: 0, max: 500 }), groundingThreshold: fc.float({ min: 0.1, max: 0.9 }), maxResponseLength: fc.integer({ min: 100, max: 5000 }), });
🧩

Composables

Combinas generadores simples (fc.string(), fc.integer()) para crear estructuras complejas (fc.record(), fc.array()). Tu dominio, tu generador.

🎯

Realistas + extremos

Mezcla inputs realistas (consultas normales) con extremos (strings vacíos, 10K chars, emojis). fc.oneof() elige aleatoriamente entre generadores.

🔄

Reproducibles

fast-check usa un seed. Si un test falla, puedes re-ejecutar con el mismo seed y reproducir el fallo exacto.

04 — Shrinking

El ejemplo mínimo que reproduce el fallo

Cuando PBT encuentra un input que viola la property, no te muestra ese input directamente — lo REDUCE al ejemplo más simple posible que aún causa el fallo. Esto se llama "shrinking" y es una de las ventajas más poderosas de PBT.

Ejemplo: Shrinking en acción

// Property: "El RAG siempre responde en menos de 5 segundos" fc.assert( fc.property(queryArbitrary, async (query) => { const start = Date.now(); const response = await rag.query(query); return (Date.now() - start) < 5000; }) ); // ❌ FALLO detectado en iteración 47 con: // Input original: "¿Cuál es el procedimiento completo detallado // paso a paso con ejemplos y referencias cruzadas de la normativa // vigente actualizada al 2025 sobre cotizaciones previsionales // obligatorias y voluntarias...?" (847 caracteres) // 🔄 SHRINKING... reduciendo el input... // Intento 1: 423 chars → falla // Intento 2: 212 chars → falla // Intento 3: 106 chars → pasa ✅ // Intento 4: 159 chars → falla // Intento 5: 133 chars → falla // Intento 6: 120 chars → pasa ✅ // Intento 7: 126 chars → falla (MÍNIMO!) // ✅ RESULTADO: El input mínimo que causa el fallo tiene 126 chars. // El RAG tarda > 5s cuando la consulta supera ~125 caracteres. // → BUG: falta chunking/truncado del input de consulta.
🔬

¿Por qué importa?

Sin shrinking: "falla con un string de 847 chars". Con shrinking: "falla con cualquier input de 126+ chars". El segundo te dice exactamente dónde está el bug.

⚙️

Es automático

fast-check hace shrinking sin que configures nada. Cada generador sabe cómo reducirse: strings se acortan, números se acercan a cero, arrays pierden elementos.

🎯

Búsqueda binaria inteligente

No reduce de uno en uno — usa búsqueda binaria para encontrar el mínimo rápidamente. De 847 chars a 126 en solo 7 intentos.

05 — Integración PBT con Kiro Specs

De correctness properties en design.md a tests automáticos

Las correctness properties que definiste en M3.1 se traducen directamente en property tests con fast-check. El agente de Kiro puede generar estos tests automáticamente desde el design.md.

Correctness Property → Property Test

// DESIGN.MD dice: // CP-1: "Para toda consulta, la respuesta incluye >= 1 fuente citada" // SE TRADUCE EN: import fc from 'fast-check'; import { queryRAG } from '../src/rag'; describe('Correctness Properties del RAG', () => { it('CP-1: Toda respuesta incluye al menos una fuente', async () => { await fc.assert( fc.asyncProperty( fc.string({ minLength: 1, maxLength: 500 }), async (query) => { const response = await queryRAG(query); // Si hay respuesta, debe tener fuentes if (response.type !== 'NO_RELEVANT_INFO') { return response.sources.length >= 1; } return true; // NO_RELEVANT_INFO es válido sin fuentes } ), { numRuns: 100 } ); }); it('CP-3: Respuesta siempre acotada', async () => { await fc.assert( fc.asyncProperty( fc.string({ minLength: 0, maxLength: 1000 }), async (query) => { const response = await queryRAG(query); return response.answer.length <= 3000 && response.answer.length >= 0; } ), { numRuns: 100 } ); }); });

Flujo: Requirements EARS → Correctness Properties (design.md) → Property Tests (fast-check) → Ejecución con 100+ iteraciones → Shrinking si falla. Todo automatizable por el agente de Kiro.

06 — Casos prácticos para el RAG

3 escenarios que PBT descubre por ti

Estos son bugs reales que PBT puede encontrar en un RAG y que los unit tests tradicionales rara vez cubren. Cada uno es un property test que vas a implementar en el workshop.

🫙

Caso 1: Consulta vacía

Property: "Para todo input (incluyendo string vacío), el RAG responde gracefully sin lanzar excepción."

PBT genera strings vacíos, solo espacios, solo newlines. ¿Tu RAG maneja "", " ", "\n\n"? ¿Retorna error 400 o explota con TypeError?

🔤

Caso 2: Encoding especial

Property: "Para todo documento con caracteres válidos UTF-8, el indexador genera embeddings sin error."

PBT genera docs con ñ, tildes, emojis 🧪, caracteres CJK, RTL. ¿El text splitter parte un carácter multi-byte a la mitad? ¿Titan Embed acepta emojis?

📊

Caso 3: Ranking estable

Property: "Para toda query que coincide con N documentos, dos ejecuciones consecutivas retornan el mismo orden de resultados."

PBT genera la misma query 2 veces. ¿El ranking es determinístico? ¿O la temperatura del LLM causa variaciones en el orden de fuentes citadas?
