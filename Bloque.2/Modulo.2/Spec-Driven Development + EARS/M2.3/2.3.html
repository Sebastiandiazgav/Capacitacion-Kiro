01 — Escribir requirements.md con EARS

Estructura de un requirements.md en Kiro

En Kiro, cada spec tiene un archivo requirements.md donde los user stories se documentan con acceptance criteria en notación EARS. El formato es Markdown estándar con una estructura predecible que el agente puede parsear y verificar.

Estructura tipo de requirements.md

# Requirements: RAG Knowledge Base ## REQ-1: Indexación de documentos nuevos [Ubiquitous] **As a** administrador del sistema **I want to** que los documentos nuevos se indexen automáticamente **So that** la KB siempre esté actualizada sin intervención manual ### Acceptance Criteria - El servicio de indexación SHALL procesar todo documento nuevo agregado a la carpeta docs/ en un máximo de 60 segundos. - El servicio de indexación SHALL generar embeddings con dimensionalidad consistente (1536 dimensiones para Titan). ## REQ-2: Consultas con fuentes citadas [Event-Driven] **As a** usuario del equipo **I want to** hacer preguntas sobre documentación interna **So that** obtenga respuestas fundamentadas con la fuente original ### Acceptance Criteria - WHEN un usuario envía una consulta al RAG, el servicio de consultas SHALL retornar una respuesta con al menos una fuente citada (nombre del documento + fragmento relevante). - WHEN un usuario envía una consulta vacía, el servicio de consultas SHALL retornar un mensaje indicando que la consulta no puede estar vacía.
🏷️

ID + Patrón EARS

Cada requirement tiene un ID único (REQ-1, REQ-2...) y una etiqueta indicando qué patrón EARS usa. Facilita trazabilidad y auditoría.

👤

User Story (As a / I want / So that)

Contexto humano: quién necesita qué y por qué. Le da al agente el "para qué" detrás del requirement técnico.

✅

Acceptance Criteria en EARS

Los criterios de aceptación usan notación EARS estricta: keywords en mayúsculas, sistema nombrado, respuesta observable y testeable.

02 — Validación automática

Kiro detecta requirements que no siguen EARS

Al crear o editar requirements.md, Kiro analiza la estructura de tus acceptance criteria. Si detecta ambigüedad, falta de keywords obligatorias, o patrones incompletos, te alerta y sugiere correcciones.

❌ Kiro detecta estos errores

❌ "El sistema debe ser rápido" → Falta SHALL, sin métrica concreta ❌ "Cuando el usuario haga algo, responder" → WHEN sin sistema nombrado, sin SHALL ❌ "El sistema should mostrar un error" → "should" no es válido, usar SHALL ❌ "Si hay error, manejar adecuadamente" → IF sin THEN, respuesta vaga ❌ "El sistema SHALL ser seguro y rápido" → Dos requirements en uno, no testeable

✅ Corrección sugerida por Kiro

✅ "El servicio de consultas SHALL responder en menos de 3 segundos (p95)." ✅ "WHEN el usuario envía una consulta, el servicio de consultas SHALL retornar una respuesta con fuentes citadas." ✅ "El servicio de autenticación SHALL rechazar tokens expirados con HTTP 401." ✅ "IF la API de embeddings retorna error, THEN el servicio de indexación SHALL reintentar 3 veces con backoff exponencial." ✅ Dividir en 2 requirements separados, cada uno con una métrica específica.
🔍

Detección de ambigüedad

Palabras como "rápido", "adecuado", "correcto" sin métrica disparan alertas. Kiro pide especificar: ¿cuántos ms? ¿qué código HTTP? ¿qué mensaje exacto?

⚠️

Keywords incorrectas

"should", "must", "will", "can" → Kiro sugiere reemplazar por SHALL. Solo SHALL indica obligatoriedad sin ambigüedad en EARS.

✂️

Requirements compuestos

Si detecta "y" conectando dos comportamientos, sugiere dividir en requirements separados. Cada EARS = un comportamiento = un test.

03 — Mapeo EARS → Design → Tasks

De EARS a correctness properties y tasks

Cada requirement EARS se transforma en una "correctness property" en el design.md (una invariante que el código debe cumplir), y luego en tasks discretas con checklist de implementación.

Pipeline de transformación

EARS → Acceptance Criteria → Correctness Property → Task → Test

📋 EARS Requirement
→
🏗️ Correctness Property (design.md)
→
☑️ Task + Checklist (tasks.md)
→
🧪 Test verificable

Ejemplo: EARS → design.md

# EARS Requirement (requirements.md): "WHEN un usuario envía una consulta al RAG, el servicio de consultas SHALL retornar una respuesta con al menos una fuente citada en un máximo de 5 segundos." # Correctness Property (design.md): ## CP-2: Respuesta con fuentes citadas - Invariante: toda respuesta del RAG incluye >= 1 fuente - Fuente = { documentName: string, chunk: string, score: number } - Tiempo de respuesta p95 < 5000ms - Si no hay chunks relevantes (score < 0.3), aplicar REQ-5 (Unwanted Behavior: responder "no encontré información") # Implicaciones de diseño: - El retriever debe retornar metadata con cada chunk - La respuesta debe incluir un array de sources[] obligatorio - Implementar timeout de 5s con circuit breaker

Ejemplo: Correctness Property → tasks.md

## Task 5: Implementar respuestas con fuentes citadas [pending] Depende de: Task 3 (retriever), Task 4 (LLM integration) ### Checklist: - [ ] El retriever retorna metadata (docName, chunk, score) - [ ] La función queryRAG() incluye array sources[] en respuesta - [ ] Si sources está vacío, retornar mensaje de "no encontré info" - [ ] Timeout de 5s implementado con circuit breaker - [ ] Test: consulta válida → respuesta con >= 1 fuente - [ ] Test: consulta sin matches → mensaje informativo - [ ] Test: timeout → respuesta graceful degradation
🔗

Trazabilidad completa

REQ-2 (EARS) → CP-2 (design) → Task 5 (implementation) → Tests derivados. Puedes rastrear cualquier línea de código hasta su requirement original.

🧪

EARS → GIVEN/WHEN/THEN

WHILE → GIVEN (setup). WHEN → WHEN (action). SHALL → THEN (assertion). Los tests se derivan automáticamente de los EARS.

☑️

Checklist = verification

Cada task tiene un checklist derivado de la correctness property. El agente verifica cada item antes de marcar la task como completed.

04 — Demo: Feature completa con EARS

De prompt a código en 3 artefactos

Flujo completo de una feature del RAG: defines la intención → Kiro genera requirements EARS → diseña la arquitectura → genera tasks → ejecuta. Cada paso tiene gate de aprobación.

Flujo de la demo

PASO 1: TU INTENCIÓN
💬 "Quiero que el RAG soporte control de acceso por rol"
PASO 2: KIRO GENERA REQUIREMENTS.MD
REQ-1 [WHERE]: Verificar permisos
REQ-2 [IF/THEN]: Sin acceso → filtrar
REQ-3 [WHEN]: Log de acceso
PASO 3: KIRO GENERA DESIGN.MD
CP-1: Middleware de permisos
CP-2: Filtrado por rol en retriever
CP-3: Audit log
PASO 4: KIRO GENERA TASKS.MD Y EJECUTA
Task 1 ✅
Task 2 ✅
Task 3 ✅
→
🎉 Feature completa

El mismo RAG que en M1 construimos en Vibe (sin plan, sin docs, sin verificación), ahora lo re-especificamos con rigor: cada comportamiento tiene un EARS, cada EARS tiene un test, cada test tiene un task que el agente ejecuta y verifica.

05 — Lenguaje natural vs EARS

¿Cuál genera mejor código?

La misma intención expresada de dos formas diferentes produce resultados radicalmente distintos cuando el agente la implementa. EARS no es burocracia — es precisión que se traduce directamente en calidad de código.

❌ Lenguaje natural libre

"El RAG debe responder rápido y citar fuentes. Si no encuentra nada, que lo maneje bien. También debería funcionar cuando hay mucha carga."

Resultado: El agente interpreta "rápido" como quiera, "manejar bien" es subjetivo, "mucha carga" sin definir. Código sin timeouts claros, sin tests precisos, comportamiento inconsistente.

✅ EARS formal

REQ-1 [Ubiquitous]: El servicio de consultas SHALL responder en < 5 segundos (p95). REQ-2 [Event-Driven]: WHEN un usuario envía una consulta, el servicio SHALL retornar respuesta con >= 1 fuente citada. REQ-3 [Unwanted]: IF no hay documentos relevantes (score < 0.3), THEN el servicio SHALL retornar "No encontré información relevante. Intente reformular su pregunta." REQ-4 [State-Driven]: WHILE el sistema procesa > 50 consultas concurrentes, el servicio SHALL mantener latencia < 5s mediante cola de prioridad.

Resultado: El agente implementa timeout de 5s, incluye sources[] obligatorio, maneja el caso vacío con mensaje específico, implementa cola bajo carga. Cada punto es testeable.
