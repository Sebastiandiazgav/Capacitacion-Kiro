# 2.3 Workshop: Re-especificar el RAG con EARS

## Objetivo

Tomar el RAG básico construido en M1 (modo Vibe, sin spec) y crear una especificación formal completa usando notación EARS. Demostrar la diferencia tangible entre construir con Vibe vs construir con Spec.

**Tiempo:** 45 minutos  
**Modo:** Spec (Requirements-First)  
**Entregable:** Spec completa con 8+ requirements EARS usando los 6 patrones

---

## Prerrequisitos

- ✅ RAG básico de M1.3 funcionando (o al menos familiarizado con él)
- ✅ Kiro instalado con credenciales AWS configuradas
- ✅ Comprensión de los 6 patrones EARS (M2.2)
- ✅ Comprensión del flujo Spec-Driven (M2.1)

---

## Paso 1: Crear la spec en Kiro (5 min)

1. Abre Kiro en tu proyecto del RAG (o crea uno nuevo)
2. En el panel de Kiro, haz clic en **+** bajo **Specs**
3. Selecciona **Feature**
4. Elige workflow **Requirements-First**
5. Describe tu feature:

```
Necesito una Knowledge Base (RAG) para documentación interna de AFP UNO.
El sistema debe indexar documentos Markdown, responder consultas con fuentes
citadas, manejar errores gracefully, soportar control de acceso por rol,
y mantener rendimiento bajo carga concurrente.
```

6. Kiro generará un borrador de requirements.md — **no lo apruebes aún**

---

## Paso 2: Escribir requirements EARS (15 min)

Edita el requirements.md generado por Kiro para incluir al menos **8 requirements** usando **todos los 6 patrones EARS**. Usa estos como guía:

### Patrón 1 — Ubiquitous (2 requirements)

```markdown
## REQ-1: Tiempo de indexación [Ubiquitous]
**As a** administrador del sistema
**I want to** que documentos nuevos se indexen rápidamente
**So that** la KB esté siempre actualizada

### Acceptance Criteria
- El servicio de indexación SHALL procesar todo documento nuevo
  en un máximo de 60 segundos desde su detección en la carpeta docs/.
- El servicio de indexación SHALL generar embeddings de 1536
  dimensiones (Titan) para cada chunk de texto.
```

```markdown
## REQ-2: Formato de respuesta [Ubiquitous]
**As a** usuario del equipo
**I want to** recibir respuestas estructuradas
**So that** pueda verificar la fuente de cada afirmación

### Acceptance Criteria
- El servicio de consultas SHALL retornar toda respuesta en formato
  JSON con los campos: answer (string), sources (array), confidence (number).
```

### Patrón 2 — Event-Driven (2 requirements)

```markdown
## REQ-3: Consulta con fuentes [Event-Driven]
**As a** usuario del equipo
**I want to** hacer preguntas sobre documentación interna
**So that** obtenga respuestas fundamentadas

### Acceptance Criteria
- WHEN un usuario envía una consulta al RAG, el servicio de consultas
  SHALL retornar una respuesta con al menos una fuente citada
  (nombre del documento + fragmento relevante) en menos de 5 segundos.
```

```markdown
## REQ-4: Re-indexación manual [Event-Driven]
**As a** administrador del sistema
**I want to** poder forzar re-indexación cuando actualizo docs
**So that** los cambios se reflejen inmediatamente

### Acceptance Criteria
- WHEN el administrador ejecuta el comando de re-indexación,
  el servicio de indexación SHALL re-procesar todos los documentos
  en la carpeta docs/ y actualizar el vector store en menos de 5 minutos.
```

### Patrón 3 — State-Driven (1 requirement)

```markdown
## REQ-5: Rendimiento bajo carga [State-Driven]
**As a** líder técnico
**I want to** que el RAG mantenga rendimiento bajo carga
**So that** el equipo pueda usarlo concurrentemente

### Acceptance Criteria
- WHILE el sistema procesa más de 20 consultas concurrentes,
  el servicio de consultas SHALL mantener tiempo de respuesta
  inferior a 8 segundos (p95) mediante cola de prioridad.
```

### Patrón 4 — Optional Feature (1 requirement)

```markdown
## REQ-6: Control de acceso por rol [Optional Feature]
**As a** administrador de seguridad
**I want to** restringir acceso a documentos sensibles
**So that** solo usuarios autorizados vean información restringida

### Acceptance Criteria
- WHERE la organización tiene habilitado el módulo de control de acceso,
  el servicio de consultas SHALL verificar el rol del usuario antes de
  incluir fragmentos de documentos restringidos en la respuesta.
```

### Patrón 5 — Unwanted Behavior (1 requirement)

```markdown
## REQ-7: Consulta sin resultados [Unwanted Behavior]
**As a** usuario del equipo
**I want to** saber cuando no hay información disponible
**So that** no reciba respuestas inventadas

### Acceptance Criteria
- IF la consulta del usuario no tiene documentos relevantes
  (score de similitud < 0.3 para todos los chunks), THEN el servicio
  de consultas SHALL retornar el mensaje "No encontré información
  relevante en la documentación. Intente reformular su pregunta o
  verifique que el tema está cubierto en los documentos indexados."
```

### Patrón 6 — Complex (1 requirement)

```markdown
## REQ-8: Acceso restringido bajo carga [Complex]
**As a** arquitecto del sistema
**I want to** que el control de acceso funcione correctamente bajo carga
**So that** no haya fugas de información por condiciones de carrera

### Acceptance Criteria
- WHILE el RAG procesa consultas concurrentes, WHEN un usuario
  consulta documentos restringidos, IF el usuario no tiene nivel
  de acceso suficiente, THEN el servicio de consultas SHALL retornar
  solo los fragmentos de documentos públicos relevantes y registrar
  el intento de acceso denegado en el audit log.
```

---

## Paso 3: Aprobar requirements y generar design (10 min)

1. Revisa los 8 requirements — asegúrate de que cada uno tiene:
   - ID único
   - Patrón EARS identificado
   - User story (As a / I want / So that)
   - Acceptance criteria con keywords EARS en mayúsculas
2. **Aprueba** los requirements en Kiro
3. Kiro generará **design.md** con:
   - Arquitectura de componentes
   - Correctness properties derivadas de cada EARS
   - Diagramas de secuencia
   - Strategy de error handling
4. Revisa y **aprueba** el design

---

## Paso 4: Generar y ejecutar tasks (10 min)

1. Kiro genera **tasks.md** con tareas discretas
2. Cada task tiene checklist derivado de las correctness properties
3. **Ejecuta las primeras 3 tasks** (clic en Run o Run All)
4. Observa:
   - ¿El agente sigue el design?
   - ¿Los tests que genera corresponden a los EARS?
   - ¿El checklist se marca al completar?

---

## Paso 5: Comparación Vibe vs Spec (5 min)

Abre lado a lado:
- El RAG de M1.3 (hecho en Vibe, sin spec)
- El RAG de este workshop (hecho con Spec + EARS)

Compara:
- ¿Cuál tiene mejor manejo de errores?
- ¿Cuál tiene tests más completos?
- ¿Cuál documenta mejor las decisiones?
- ¿Cuál sería más fácil de mantener por otro developer?

---

## Entregable

Al final del workshop debes tener:

- [ ] Spec completa en `.kiro/specs/rag-knowledge-base/`
- [ ] `requirements.md` con 8+ requirements usando los 6 patrones EARS
- [ ] `design.md` con correctness properties derivadas
- [ ] `tasks.md` con checklist de implementación
- [ ] Al menos 3 tasks ejecutadas exitosamente
- [ ] Comparación documentada: Vibe vs Spec
