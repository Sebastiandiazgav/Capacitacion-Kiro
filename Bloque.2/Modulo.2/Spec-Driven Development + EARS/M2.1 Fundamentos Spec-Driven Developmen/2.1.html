01 — Filosofía

"Diseña primero, implementa después"

En desarrollo tradicional con IA, el developer escribe un prompt y el agente genera código inmediatamente. Es rápido, pero frágil. Spec-Driven Development invierte el proceso: primero defines qué necesitas (requirements), luego cómo se construye (design), y finalmente el agente ejecuta un plan concreto (tasks).

"La unidad de trabajo en Kiro no es un prompt suelto — es una especificación estructurada que el agente usa para planificar, implementar, verificar y documentar tu feature de punta a punta."

— Kiro, spec-driven development philosophy
🏗️

Analogía: Arquitecto vs Albañil

Un arquitecto no construye sin planos. Un developer no debería codear sin spec. Los planos son el contrato — la spec es tu contrato con el agente.

📋

Spec = Contrato verificable

Cada spec define acceptance criteria claros. El agente verifica automáticamente que cada tarea cumple con lo especificado antes de marcarla como completada.

🔍

Trazabilidad total

Cada línea de código tiene origen: viene de una task, que viene de un design, que viene de un requirement. Puedes rastrear cualquier decisión hasta su origen.

👥

Colaboración equipo-IA

Tú defines el qué y el por qué. El agente propone el cómo. Tú apruebas cada fase antes de avanzar. Nunca pierdes el control.

02 — El flujo completo

Requirements → Design → Tasks

Define qué necesitas. User stories con acceptance criteria en notación EARS. Cada requirement es verificable y medible.

03 — Estructura en el proyecto

.kiro/specs/ — Donde viven las specs

Cada feature tiene su propia carpeta dentro de .kiro/specs/. Los 3 artefactos se generan automáticamente y quedan versionados con tu código. Son archivos Markdown — legibles, editables y portables.

Estructura de archivos

tu-proyecto/ ├── .kiro/ │ └── specs/ │ ├── autenticacion-oauth/ │ │ ├── requirements.md ← User stories + acceptance criteria │ │ ├── design.md ← Arquitectura + diagramas │ │ └── tasks.md ← Tareas ejecutables │ ├── modulo-cotizaciones/ │ │ ├── requirements.md │ │ ├── design.md │ │ └── tasks.md │ └── dashboard-reportes/ │ ├── requirements.md │ ├── design.md │ └── tasks.md ├── src/ ├── package.json └── ...
📄

requirements.md

User stories escritas en notación EARS (Easy Approach to Requirements Syntax). Cada historia tiene ID, descripción y acceptance criteria verificables.

📐

design.md

Documento técnico con arquitectura de componentes, diagramas de secuencia (Mermaid), interfaces/tipos, data flow, estrategia de error handling y testing.

☑️

tasks.md

Lista numerada de tareas discretas. Cada tarea tiene un outcome claro. Se ejecutan secuencialmente o en paralelo según su dependency graph.

Ejemplo: requirements.md

# Requirements: Módulo de Cotizaciones ## REQ-1: Recepción de planillas **As a** operador de recaudación **I want to** recibir planillas de cotizaciones vía portal empleador **So that** las cotizaciones se procesen automáticamente ### Acceptance Criteria - GIVEN un archivo previred válido (CSV/XML) WHEN se sube al portal THEN se validan RUT, montos y períodos en < 30 segundos - GIVEN un archivo con errores parciales WHEN se procesa THEN se genera reporte de rechazo con detalle por fila
04 — Feature Workflows

Tres formas de iniciar una spec

Dependiendo de tu situación (¿ya tienes diseño? ¿solo una idea? ¿feature simple?), Kiro ofrece tres workflows para crear specs.

Más común

📋 Requirements-First

Defines qué necesitas en lenguaje natural. El agente genera requirements formales → luego design → luego tasks.

  • Ideal cuando sabes el "qué" pero no el "cómo"
  • El agente propone la arquitectura
  • Tú apruebas cada fase antes de avanzar
  • Máximo control sobre el proceso
Si ya tienes diseño

🏗️ Design-First

Ya tienes arquitectura (whiteboard, diagrama, doc previo). Subes el diseño y el agente extrae requirements + genera tasks.

  • Sube diagramas (PNG, JPG) o pega contenido de diseño
  • El agente formaliza tu diseño en design.md
  • Deriva requirements del diseño automáticamente
  • Ideal post-reunión de arquitectura
Fast track

⚡ Quick Plan

Auto-genera los 3 artefactos sin gates de aprobación. Para features simples y bien entendidas donde no necesitas iterar.

  • Un solo prompt genera todo
  • Sin paradas de aprobación
  • Ideal para features pequeñas y claras
  • Puedes editar los artefactos después

¿Cuándo usar cada uno?
Requirements-First → feature nueva, exploratoria
Design-First → ya hiciste spike o whiteboarding
Quick Plan → feature obvia, CRUD simple, cambio menor

05 — Ejecución de Tasks

El agente como ejecutor de tasks

Una vez que apruebas el tasks.md, el agente ejecuta cada tarea de forma autónoma. Verifica resultados entre cada paso. Puedes ejecutar una tarea individual o lanzar todas — las independientes corren en paralelo.

Ejecución paralela con Dependency Graph

Kiro analiza dependencias y agrupa tareas en waves

WAVE 1 — Tareas sin dependencias (paralelo)
Task 1: Setup proyecto
Task 2: Crear interfaces
Task 3: Config DB
WAVE 2 — Dependen de Wave 1 (paralelo)
Task 4: Implementar API
Task 5: Implementar validaciones
WAVE 3 — Dependen de Wave 2
Task 6: Tests de integración
→
✅ Feature completa
▶️

Run individual

Ejecuta una sola tarea. Ideal para testing granular o cuando quieres revisar paso a paso.

⏩

Run All Tasks

Kiro analiza dependencias, agrupa en waves y ejecuta en paralelo. Reduce tiempos significativamente.

📊

Status Tracking

Cada tarea tiene estado en tiempo real: pending → in_progress → completed. Ves el progreso sin salir del IDE.

🔄

Verificación automática

Entre cada tarea, el agente verifica que el resultado cumple con el diseño y los acceptance criteria antes de marcar como completada.

Ejemplo: tasks.md

# Tasks: Módulo de Cotizaciones ## Task 1: Setup del proyecto [completed] - Crear estructura de carpetas src/cotizaciones/ - Configurar dependencias necesarias - Crear archivo de configuración de la DB ## Task 2: Definir interfaces TypeScript [completed] - Crear tipos para Cotizacion, Planilla, ResultadoValidacion - Exportar desde src/cotizaciones/types.ts ## Task 3: Implementar validador de planillas [in_progress] - Implementar validación de RUT (módulo 11) - Validar montos dentro de rango legal - Validar períodos no duplicados - Generar reporte de errores por fila ## Task 4: Implementar servicio de acreditación [pending] - Depende de: Task 2, Task 3 - Acreditar cotizaciones válidas en cuenta individual - Distribuir según fondo elegido por afiliado
06 — El anti-patrón

Vibe directo sin spec = deuda técnica garantizada

El modo Vibe es excelente para explorar y prototipar. Pero usarlo para construir features de producción sin una spec es como construir un edificio sin planos: funciona al principio, pero colapsa cuando escala.

❌ Vibe directo (sin spec)

  • No hay documentación de decisiones de diseño
  • Código sin trazabilidad — ¿por qué se hizo así?
  • El agente interpreta diferente en cada sesión
  • Sin acceptance criteria = sin forma de verificar
  • Refactoring doloroso — no sabes qué rompes
  • Onboarding de nuevos devs es un misterio
  • Prototipos que nunca escalan a producción
  • Iteraciones infinitas corrigiendo output de la IA

✅ Con Spec-Driven

  • Cada decisión documentada en design.md
  • Trazabilidad completa: requirement → task → código
  • El agente sigue un plan consistente siempre
  • Acceptance criteria claros y verificables
  • Refactoring seguro — sabes el contrato
  • Onboarding: lee la spec y entiende la feature
  • Código production-ready desde el primer task
  • Una sola iteración con verificación automática

Regla de oro: Usa Vibe para explorar y aprender. Usa Spec para construir features reales que van a producción. Nunca al revés.

💬

¿Cuándo usar Vibe?

Prototipos rápidos, exploración de APIs, preguntas al agente, spikes técnicos, aprendizaje. Nada que vaya directo a producción.

📋

¿Cuándo usar Spec?

Features nuevas, bugfixes complejos, cualquier cosa que el equipo deba mantener, código que va a producción, funcionalidad con acceptance criteria.

🎯

La transición

Puedes explorar en Vibe y cuando encuentras el enfoque correcto, crear una Spec para implementar con rigor. No son excluyentes — son complementarios.

07 — Resumen ejecutivo

Lo que debes recordar de Spec-Driven Development

"Diseña primero, implementa después"

La spec es el contrato entre tu intención y la ejecución del agente. Sin spec, el agente adivina. Con spec, el agente ejecuta un plan verificable.

Tres fases: Requirements → Design → Tasks

Cada fase genera un artefacto Markdown. Entre cada fase hay un gate de aprobación — tú revisas y decides antes de que el agente avance.

Los artefactos viven en .kiro/specs/<feature>/

requirements.md, design.md, tasks.md — versionados con tu código, legibles por humanos, parseables por el agente.

Tres workflows: Requirements-First, Design-First, Quick Plan

Requirements-First si sabes el "qué". Design-First si ya tienes arquitectura. Quick Plan para features simples sin gates.

Ejecución paralela con dependency graph (Waves)

El agente agrupa tasks independientes y las ejecuta en paralelo. Verificación automática entre cada wave. Status tracking en tiempo real.

Vibe para explorar, Spec para construir — nunca al revés

Vibe es para prototipos y aprendizaje. Spec es para features de producción. Son complementarios, no excluyentes.
