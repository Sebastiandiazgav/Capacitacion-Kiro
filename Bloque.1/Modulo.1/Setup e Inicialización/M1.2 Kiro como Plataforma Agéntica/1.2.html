Kiro como
Plataforma Agéntica

Kiro no es un IDE con autocompletado inteligente. Es una plataforma donde agentes autónomos analizan, planifican, implementan y verifican — mientras tú supervisas y tomas las decisiones de diseño.

Explorar →
01 — El cambio de paradigma

De Desarrollo Imperativo a Desarrollo Dirigido por Intención

Los IDEs tradicionales con IA (Copilot, Cursor) siguen un modelo imperativo: tú escribes, la IA completa. Kiro introduce el "Intent-Driven Development" — defines qué quieres lograr y los agentes ejecutan el cómo, con rigor de ingeniería.

"La unidad de trabajo en Kiro no es un prompt — es una especificación estructurada que el agente usa para planificar, implementar, verificar y documentar tu feature de punta a punta."

— DZone, "The Agentic IDE That Makes Specs the Unit of Work" (2026)
Aspecto IDE con IA (Copilot, Cursor) Plataforma Agéntica (Kiro)
Modelo de interacción Tú escribes → IA completa Tú defines intención → Agente ejecuta plan completo
Unidad de trabajo Línea de código / prompt suelto Especificación (requirements + design + tasks)
Planificación ✗ No planifica, responde ✓ Genera plan antes de codificar
Verificación ✗ Manual ✓ Automática entre cada tarea
Documentación ✗ Separada del flujo ✓ Generada y sincronizada automáticamente
Autonomía Reactiva (espera tu input) Proactiva (ejecuta tareas end-to-end)
Sub-agentes ✗ No existe ✓ Delega en agentes especializados en paralelo
02 — Arquitectura interna

Agente Principal → Sub-agentes → Herramientas

Kiro opera con una jerarquía de agentes. El agente principal (orquestador) recibe tu intención, construye un plan, y delega tareas a sub-agentes especializados. Cada sub-agente tiene acceso a herramientas específicas y reporta resultados al orquestador.

Flujo de orquestación agéntica

El agente principal coordina, los sub-agentes ejecutan, las herramientas interactúan con el mundo

🧠

Agente Principal

El orquestador central. Recibe tu intención, la descompone en tareas, decide qué sub-agentes necesita, coordina la ejecución y verifica resultados antes de entregarte el output final.

🤖

Sub-agentes

Agentes especializados que se ejecutan en paralelo. Cada uno tiene un contexto reducido y enfocado — no se distraen con información irrelevante. Reportan al orquestador.

🔧

Herramientas (Tools)

Los sub-agentes usan herramientas para interactuar con el mundo: leer/escribir archivos, ejecutar comandos, buscar código, hacer requests HTTP, conectar con APIs vía MCP.

🔌

MCP (Model Context Protocol)

Protocolo estándar que conecta a los agentes con servicios externos: bases de datos, APIs, documentación, herramientas de deploy. Extiende capacidades sin límites.

🧭

Steering (Directrices)

Archivos en .kiro/steering/ que inyectan reglas persistentes: convenciones de código, estándares del equipo, guías de arquitectura. Los agentes las respetan siempre.

⚡

Hooks (Event-Driven)

Automatizaciones que se disparan ante eventos: al guardar un archivo, al completar una tarea, antes de ejecutar una herramienta. Los agentes reaccionan sin que tú intervengas.

03 — El ejército de agentes

Cada developer tiene múltiples agentes especializados

En Kiro, un developer no trabaja con "un solo asistente genérico". Puede configurar un ecosistema de agentes especializados — cada uno con su propio contexto, herramientas permitidas y reglas de comportamiento. Son portables, versionables y compartibles con el equipo.

🔍

Agente de Code Review

Revisa PRs automáticamente. Conoce los estándares del equipo, detecta anti-patterns y sugiere mejoras con contexto.

🏗️

Agente de Arquitectura

Evalúa decisiones de diseño, genera diagramas, verifica que el código nuevo sea consistente con la arquitectura existente.

🧪

Agente de Testing

Genera tests unitarios, de integración y property-based. Detecta edge cases que los tests manuales no cubren.

📖

Agente de Documentación

Mantiene docs sincronizadas con el código. Genera README, API docs y changelogs automáticamente al implementar cambios.

☁️

Agente DevOps/Cloud

Gestiona infraestructura como código, genera Terraform/CDK, troubleshoots producción y automatiza deploys.

🔒

Agente de Seguridad

Escanea vulnerabilidades, valida dependencias, detecta secrets expuestos y genera reportes de compliance.

📂 Configuración de un agente custom

Los agentes se definen en .kiro/agents/ como archivos Markdown con front-matter:

# .kiro/agents/backend-specialist.md --- name: Backend Specialist description: Agente experto en backend Node.js/Python tools: allow: - "@builtin" # Todas las herramientas built-in - "mcp:database" # MCP server de base de datos deny: - "fs_write" # No puede escribir fuera de src/ resources: - src/api/** - server.js - docs/backend-standards.md --- ## Rol Eres un especialista en desarrollo backend. Solo trabajas con archivos en src/api/ y server.js. Sigues las convenciones de backend-standards.md. ## Reglas 1. Siempre usa async/await 2. Manejo de errores con try/catch en cada endpoint 3. Validación de inputs con Zod/Joi 4. Documentación JSDoc en cada función pública

"Creas un agente como configuración en .kiro/agents/ y lo ejecutas bajo demanda donde quieras — terminal local, step de pipeline, o servidor con cron."

04 — Kiro CLI

Agentes en tu terminal, en cualquier lugar

Kiro CLI trae los mismos agentes del IDE directamente a tu terminal. No es un servicio corriendo 24/7 — es un proceso que se invoca cuando lo necesitas, con el contexto de tu proyecto. Tu carpeta .kiro/ funciona en ambos entornos: configuras una vez, usas en todas partes.

🖥️

Terminal interactiva

Ejecuta kiro-cli y chatea con agentes directamente en tu terminal. Scaffold proyectos, debug issues, escribe IaC — sin salir del shell.

🤖

Headless Mode (CI/CD)

Genera un API key, configúralo como variable de entorno, y Kiro CLI se ejecuta programáticamente. Ideal para pipelines, scripts de build y automatización.

🪟

Multiplataforma

Disponible para macOS, Linux y Windows (nativo desde CLI 2.0, abril 2026). Sin necesidad de WSL.

Instalación y uso

# Instalar Kiro CLI $ curl -fsSL https://cli.kiro.dev/install | bash # Iniciar chat interactivo $ kiro-cli # Ejecutar un agente custom $ kiro-cli --agent backend-specialist # Headless: ejecutar en pipeline sin interacción $ KIRO_API_KEY=xxx kiro-cli --headless \ --prompt "Genera PR con los cambios de la rama feature/auth"

Autenticación flexible

05 — Los 3 modos de trabajo

Vibe · Spec · AI-DLC

Kiro ofrece tres modos que escalan en rigor. Cada uno tiene su momento y propósito. Hoy usamos Vibe para resultados inmediatos. A partir del Módulo 2, Spec. En el Módulo 11, el ciclo completo AI-DLC.

Activo hoy — M1

💬 Vibe (Chat Agéntico)

Exploración libre, prototipos rápidos, preguntas al agente. Conversación fluida sin estructura formal. Resultados inmediatos.

  • Prototipado sin compromiso
  • Preguntas exploratorias sobre el código
  • Generación rápida sin plan previo
  • Ideal para aprender y experimentar
  • Se usa cuando no necesitas trazabilidad
📍 Este workshop (Módulo 1)
Módulo 2 en adelante

📋 Spec (Desarrollo Estructurado)

Desarrollo con especificación formal: requirements → design → tasks. El agente planifica antes de codificar.

  • Requirements con EARS notation y acceptance criteria
  • Design con diagramas de secuencia y arquitectura
  • Tasks discretas, ejecutables y verificables
  • Ejecución en paralelo con dependency graph
  • Quick Plan para features simples
  • Bugfix Specs para bugs con prevención de regresiones
📍 A partir del Módulo 2
Módulo 11

🔬 AI-DLC (AI-Driven Development Life Cycle)

Ciclo completo y riguroso de desarrollo con IA. Máximo control, documentación y trazabilidad para producción enterprise.

  • Ciclo completo: ideación → spec → implementación → testing → deploy
  • Auditoría y trazabilidad de cada decisión del agente
  • Compliance automático (SOC 2, HIPAA)
  • Integración con flujos de aprobación del equipo
  • Rigor de producción enterprise
📍 Se verá en Módulo 11

Vibe es para explorar · Spec es para construir con calidad · AI-DLC es para rigor de producción

06 — Resumen ejecutivo

Lo que debes recordar de esta sección

Kiro no es un IDE con IA — es una plataforma agéntica

La diferencia clave es autonomía: los agentes planifican, ejecutan y verifican de forma independiente. Tú supervisas y tomas decisiones de diseño.

Arquitectura jerárquica: Orquestador → Sub-agentes → Herramientas

El agente principal coordina. Los sub-agentes se especializan y ejecutan en paralelo. Las herramientas (MCP, terminal, file system) interactúan con el mundo.

Cada developer tiene su propio ejército de agentes

Configurados en .kiro/agents/, son portables, versionables y compartibles. Cada agente tiene contexto, herramientas y reglas propias.

Kiro CLI: mismos agentes, en cualquier lugar

Terminal local, GitHub Actions, cron, pipelines CI/CD. Con headless mode puedes automatizar completamente sin intervención humana.

Tres modos que escalan en rigor

Vibe (hoy) → Spec (M2+) → AI-DLC (M11). De exploración libre a rigor enterprise.

Spec-Driven Development: la unidad de trabajo es la especificación

No un prompt suelto. Requirements + Design + Tasks — con verificación automática entre cada paso y ejecución paralela.
